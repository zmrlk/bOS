# bOS -- Self-Evolution 2.0

## Philosophy

bOS is not static software -- it is a **living system** that grows with the user. It does not just remember -- it REFLECTS, LEARNS, DISCOVERS, and IMPROVES. Experts stay sharp. bOS evolves through five phases: health check, intelligence tuning, skill expansion, impact measurement, and fresh data delivery. Every proposed change must pass an **Objective Kernel** -- alignment with the user's goals, constraints, and values.

---

## When This Runs

### Automatic triggers (proactive, via @boss):
- **Weekly** (Sunday evening, before /plan-week): Lightweight scan (Phase 1 quick + Phase 2 quick)
- **After 2 weeks of use**: First deep evolution (all 5 phases)
- **Monthly** (during maintenance): Full evolution cycle (all 5 phases)
- **After major life change** (new job, new goal, new pack activated): Targeted discovery
- **After 3+ negative feedback signals on any agent**: Auto-trigger Phase 2 for that agent

### Manual trigger:
- User says `/evolve` or "improve yourself" / "what's new" / "upgrade" / "optimize"

---

## Objective Kernel (gates ALL proposals)

Every proposal generated by any phase MUST pass this alignment check before being presented to the user. No exceptions.

```
ALIGNMENT CHECK:
1. PURPOSE  -- Does this serve primary_goal? (from profile.md)
2. BUDGET   -- Does it respect financial constraints? (buffer, discretionary)
3. CAPACITY -- Does it fit within available_hours? (check with @coo)
4. HEALTH   -- Does it violate any health/wellness constraint?
5. VALUES   -- Does it align with user values? (communication_style, work_style)
6. SAFETY   -- Does it pass security requirements? (score >=8 for tools)
```

Result: PASS (all 6 green) | CONDITIONAL (1-2 yellow, flag which) | BLOCK (any red, do not present).

## Evolution Cycle (5 phases)

### Phase 1: SYSTEM HEALTH -- Infrastructure, MCPs, Security

**This phase ensures the foundation is solid before optimizing the intelligence layer.**

```
[1/5] System Health...
```

#### 1A. Skill Audit -- Match skills to user profile

Map `user_type` + `active_packs` + `primary_goal` to skill categories:

| Profile | Critical Skills | Nice-to-have |
|---------|----------------|-------------|
| Freelancer + Business | /invoice, /proposal, /competitive, /verify, /timetrack | /design, /repurpose |
| Developer + Business | /code, /analyze, /invoice | /api, /deploy (custom) |
| Manager/Leader | /standup, /sprint, /task | /1-1, /feedback (custom) |
| Student + Learning | /learn-path, /focus, /reflect | /study (custom) |
| Employee + Career | /goal, /decide, /network | /cv (custom) |
| Health pack active | /log-workout, /habit, /energy-map | /meal-plan (custom) |
| Life pack active | /budget, /habit, /reflect, /focus | /money-flow |

**Steps:** Read `.claude/skills/` -> compare to profile needs -> check marketplace registry (`skills-registry.json` via git) -> cross-reference `active_packs`, `active_agents` -> check `state/marketplace.md` for installed.

**Present:** Marketplace (1-click) + Custom (buildable) + Recommended (not installed). AskUserQuestion: "Install recommended" / "Let me pick" / "Create custom" / "Skip".

**Fallback:** Registry unavailable -> `git show origin/main:.claude/skills/` -> fetch and install.

#### 1B. MCP Discovery -- Search for new capabilities

Batch WebSearch based on profile.md: user's tools without MCP, packs with gaps, aggregators (mcp.so, glama.ai/mcp), social platforms in `active_platforms` without MCPs.

**Security filter (MANDATORY) -- every MCP must pass ALL 8:** GitHub >=100 stars OR marketplace listing | last commit <3 months | positive reviews on >=2 sources | no sudo/root | open source | clear permissions | no security red flags | known author.

**Scoring:** 10 (official + >1000 stars) | 9 (>500 + active) | 8 (>100 + clean) | <8: REJECT.
**File date filter:** >365d -> skip. 90-365d -> ask. <90d -> active.
**Typosquatting check:** npm name must match GitHub repo EXACTLY.

#### 1C. MCP Re-Verification -- Check existing connections

For each MCP in profile.md -> `Connected MCPs`: test connectivity, check for updates (WebSearch changelog), check usage (unused 60+ days -> flag), re-score security (score <8 -> warn).

**Security Incident Response:** CVE/advisory/compromise found -> ALERT user immediately -> offer disconnect -> search replacement -> log incident.

### Phase 2: AGENT INTELLIGENCE -- Reflexion, Prompt Optimization, Routing

**This is the brain of evolution. bOS analyzes its own performance and tunes itself.**

```
[2/5] Agent Intelligence...
```

#### 2A. Reflexion Analysis -- Cross-agent pattern mining

**Data sources:**
- Agent memory files (`~/.claude/agent-memory/[agent]/`) -- read MEMORY.md from each active agent
- Micro-feedback history (from /review-week and inline feedback)
- Context-bus signals (especially `calibration` type entries)
- State files for outcome data (tasks completion, goal progress, habit streaks)

**Analysis steps:**
1. **Per-agent scan:** Extract positive patterns (engagement, completions), negative patterns (corrections, abandonment, re-asks), blind spots (asked about but no data for)
2. **Cross-agent patterns:** Same complaint across agents? Conflicting advice? Unacknowledged signals?
3. **Interaction quality scoring** (per agent, 1-10): 10=followed+positive | 7-9=effective+minor fixes | 4-6=mixed | 1-3=frequent overrides

**Output:** Agent Performance Map (stored in `state/evolution-proposals.md`).

#### 2B. Self-Evolving Prompts -- Automated agent improvement

**Trigger:** An agent accumulates 3+ negative signals (corrections, overrides, "nietrafione" feedback, abandoned interactions) within a 30-day window.

**Process:**
1. **Collect evidence:** Gather negative interactions -- what user asked, what agent said, how user corrected, any explicit feedback
2. **Classify failure:** `tone_mismatch` | `context_miss` | `overreach` | `bad_default` | `stale_knowledge` | `wrong_priority`
3. **Generate prompt patch:** Targeted instruction change with evidence, before/after, expected improvement
4. **Version agent file:** Copy `.claude/agents/[agent].md` -> `state/.backup/agents/[agent]-v[N].md` (increment N)
5. **Store proposal:** Append to `state/evolution-proposals.md` (see schema below)
6. **Present to user** with before/after comparison (adapted to tech_comfort)

**For "not technical" users:**
```
  I noticed @[agent] keeps [problem in plain language].
  Examples: [1-2 simple examples].
  I can adjust how [agent] works so it [improvement].
  Want me to apply this fix?
```

**For "I code" users:**
```
  PROMPT PATCH: @[agent] (3 failures in 30d)
  Pattern: [category]
  Evidence: [examples]
  BEFORE: "[current instruction excerpt]"
  AFTER:  "[proposed instruction]"
  Apply? (yes / modify / skip)
```

**Rollback:** If after applying a patch, negative feedback continues for 2 more interactions -> auto-rollback to previous version, flag for manual review.

#### 2C. Routing Optimization -- Fix misroutes

**Data sources:**
- Context-bus entries where agents redirected to another agent
- User corrections ("I meant @X not @Y", or re-asking the same question to a different agent)
- Agent memory notes about out-of-domain requests

**Analysis:**
1. Count misroutes per agent pair (e.g., sent to @ceo but should have been @mentor: 3x)
2. Identify missing disambiguation rules
3. Check if any natural language triggers are routing to the wrong skill

**Output:**
```
  ROUTING ISSUES DETECTED:
  - "[phrase]" routed to @[wrong] 3x, should go to @[correct]
  - @[agent1] and @[agent2] overlap on [topic] -- no disambiguation rule

  PROPOSED FIX:
  Add to CLAUDE.md disambiguation table:
  | "[topic]" | @[correct] | @[wrong] |
```

Present to user for approval. If approved -> update CLAUDE.md routing table.

### Phase 3: SKILL EVOLUTION -- Patterns, Knowledge, Custom Creation

**Combines existing pattern detection with intelligence-driven skill proposals.**

```
[3/5] Skill Evolution...
```

#### 3A. Knowledge Freshness

- Profile.md fields with dates -> flag anything >3 months old
- Agent calibrations -> flag stale entries (>60 days no update)
- Pricing/tools in agent memory -> verify current accuracy via WebSearch

#### 3B. State Data Verification

- Goals without progress >3 months -> flag for @coach
- Pipeline leads without update >30 days -> flag for @sales
- Financial data from >2 months ago -> flag for @finance/@cfo

#### 3C. Skill Pattern Detection

Check agent memory + state files for repeated user actions:
- Action done >3x/week manually -> candidate for new skill
- Same multi-step process repeated -> package as skill
- Repeated complaints about a gap -> create skill to fill it

**Detection logic (expanded):**
1. Scanned files: CSV/Excel with client data -> `/sales-report`. Code repos -> `/deploy`.
2. Active tools: Notion -> `/sync-notion`. CRM -> `/crm-sync`.
3. Pain points from memory: "always forget follow-up" -> `/follow-up`.

**When creating a custom skill:**
1. Generate SKILL.md with proper frontmatter
2. Define: trigger words, steps, state file integration, agent routing
3. Present blueprint via AskUserQuestion: "Create" / "Modify first" / "Skip"
4. If approved -> save to `.claude/skills/[name]/SKILL.md`
5. Register in natural language routing
6. Monitor for 2 weeks -> if unused, offer to remove

#### 3D. Intelligence-Driven Skill Proposals (NEW)

Based on Phase 2 reflexion findings, propose skills that address systemic gaps:
- Agent frequently asked about topic with no skill -> create dedicated skill
- Cross-agent coordination weak in an area -> create orchestration skill
- User's primary_goal not well-served by existing skills -> propose targeted skill

**Pass through Objective Kernel before presenting.**

#### 3E. Tool & Integration Discovery

- Missing calendar/email integration -> find and suggest
- Tools bOS does not know about -> research, find MCP or workaround
- Social platform MCPs (same discovery patterns as v1)

## Lightweight Scan (weekly checks)

Not every evolution needs a full cycle. Weekly checks (Sunday evening):

1. **Quick MCP health:** Test call to each connected MCP
2. **Quick pattern check:** New repeated actions in last 7 days?
3. **Stale data check:** Profile fields or calibrations >60 days old?

Notable finding -> surface in /plan-week: "During my weekly check, I found [X]. Want me to look deeper? Say /evolve."
Nothing found -> log silently.

## Rules

1. **Always ask before installing or modifying.** Discovery != action.
2. **Show your sources.** Every recommendation backed by evidence.
3. **Max 5 suggestions per cycle.** Quality > quantity.
4. **Respect rejections.** Rejected twice -> stop suggesting that category.
5. **Security first.** Score <8 = reject. No exceptions.
6. **Do not fix what works.** Smooth workflow -> do not disrupt.
7. **Token awareness.** Full cycle is heavy. Warn: "This will use more resources." Adapt to tech_comfort.
8. **Log everything.** Every cycle logged for learning.
9. **Re-verify regularly.** Connected != safe forever. Monthly re-checks mandatory.
10. **Adapt language.** All output in user's language. Search queries stay English.
11. **Always show price.** Paid suggestions MUST include cost. Unclear -> research first.
12. **Cross-domain impact.** Money -> @cfo/@finance. Health -> @wellness. Capacity -> @coo.
13. **Objective Kernel gates everything.** No proposal reaches the user without passing alignment.
14. **Version before modifying.** Agent files always backed up before prompt patches.
15. **Auto-rollback on failure.** Prompt patch makes things worse -> revert automatically.
16. **Anti-re-discovery.** Before presenting ANY proposal, check: agent memory, context-bus, recent conversation context, state files. If the issue was already discussed or established → reference it ("Jak ustaliliśmy...") instead of presenting as new discovery. /evolve discovers GAPS, not things the system already knows.
